# Skill: event-processing-depth-log-detect

## 触发条件
当系统在 user_message 或 reflect 事件触发后，未记录任何具体的分析结果、行动日志或关键输出时

## 行为规则
## 1. 检测方法
- 检查 user_message 和 reflect 事件处理流程中，在事件触发后是否有对应的分析结论、提取的信息、执行的行动或关键输出日志。若日志中仅记录事件触发（如“收到消息”、“开始反思”）而无任何有效负载（如意图识别结果、实体提取、情感评分、反思结论、下一步计划、行动指令或“无操作”状态），则判定为缺失深度处理。
- 对比预期处理流程与实际日志，定位缺失环节。

## 2. 修复建议
- 在每个事件处理节点后，强制添加至少一条结构化日志，记录事件ID、触发时间、分析摘要、行动指令或输出结果。若分析后无行动，也应记录“无操作”状态。
- 优化事件处理流程，确保在用户消息处理后至少输出一条分析性记录（如意图识别、实体提取、情感评分等）。
- 在 reflect 事件中，要求必须输出反思结论或下一步计划，避免空日志。
- 增加事件处理链路的中间步骤日志，例如在解析、决策、执行各阶段分别记录，便于定位缺失环节。
- 添加超时或空结果重试机制。
- 对未处理的消息进行告警或转入人工处理。

## 3. 相关案例
- 用户发送消息后，系统仅记录“收到消息”而未记录意图识别结果或回复内容。修复后输出类似 `intent: help_request` 的分析记录。
- reflect 事件触发后，仅记录“开始反思”而未记录反思结论或改进建议。修复后输出类似 `reflect_result: no_action_required` 的结论。
- 用户发送 'help' 后，日志仅记录 'user_message received'，未输出任何意图分析结果。增加意图解析步骤后，日志输出 'intent: help_request'。
- reflect 事件触发后无任何输出，原因是反思函数未实现。实现后日志输出 'reflect_result: no_action_required'。
- 用户发送消息后无任何响应日志，原因是意图匹配失败后未记录，增加默认匹配分支后解决。
- reflect 后无改进动作，原因是反思结果未传递给执行模块，修复数据流后正常。
