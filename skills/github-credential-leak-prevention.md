# Skill: github-credential-leak-prevention

## 触发条件
在分析 GitHub 提交历史或当前仓库文件时，检测到包含硬编码的敏感信息（如用户名、密码、API 密钥、本地路径等）被提交到版本控制中。

## 行为规则
1. **检测方法**：
   - 扫描提交文件和仓库内容，使用正则表达式匹配常见敏感信息模式，包括但不限于：
     - Windows 用户名和绝对路径：如 `<你的用户名>`、`C:\Users\...`、`C:/Users/\w+`
     - 凭据类：`password\s*[:=]\s*\S+`、`api[_-]?key\s*[:=]\s*\S+`、`token\s*[:=]\s*\S+`
   - 检查 `.gitignore` 中是否遗漏了常见的敏感文件类型（如 `.env`、`*.key`、`config.local.*`、`settings.json` 等）。
   - 标记包含敏感信息的文件路径和具体行号。

2. **修复建议**：
   - 立即从 Git 历史中移除敏感文件或内容，推荐使用 `git filter-branch` 或 BFG Repo-Cleaner 清理提交历史。
   - 将硬编码的敏感信息替换为环境变量（如 `%USERNAME%`）或配置文件模板，并确保凭据已修改或撤销。
   - 在 `.gitignore` 中添加配置文件路径，防止未来提交；推荐使用密钥管理服务（如 Vault）或环境变量管理敏感信息。
   - 配置 `git-secrets`、`pre-commit` 钩子等工具，在提交前自动拦截敏感信息。

3. **相关案例**：
   - 用户在 `settings.json` 中硬编码了 Windows 用户名，导致个人信息暴露在公开仓库中，需立即清理提交历史并更新凭证。
   - 经验 ID 92, 72 均涉及敏感信息提交到版本控制，需确保规则覆盖更全面的敏感信息类型（如密钥、令牌、本地路径等）。
