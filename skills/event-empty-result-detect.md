# Skill: event-empty-result-detect

## 触发条件
当系统执行 periodic_check、user_message 或 reflect 等关键动作后，日志中未记录任何有效结果、分析数据或产出数据为空时触发。若连续多次（如3次以上）结果均为空，则判定触发。

## 行为规则
## 1. 检测方法
- 检查动作执行后的日志输出，确认是否存在结果记录。
- 对比动作输入和输出，验证是否缺失关键字段。
- 检查日志收集流程是否完整，确认是否因日志级别过滤、采集中断、异步或缓存等原因导致结果未被记录或丢失。
- 若连续多次（如3次以上）结果均为空，则判定为异常。

## 2. 修复建议
- 审查动作实现代码，确保每个事件处理函数在完成处理后返回或记录至少一个有效字段（如分析结论、行动记录、状态变更等）。
- 在动作执行关键节点（如用户输入解析后、反思完成后）增加日志输出，记录中间结果或关键字段。即使无具体内容也应输出占位符（如 'no_action_taken'）。
- 检查日志收集管道是否正常，确认日志是否被正确写入、传输和存储，避免数据丢失或截断。
- 对空结果情况添加默认处理或异常告警，如增加异常捕获并记录错误。

## 3. 相关案例
- 经验[50]：所有 periodic_check 和 user_message 的结果均为空，需检查动作实现或日志收集流程。
- 经验[40]：user_message 和 reflect 触发后未记录任何分析结果或行动，需优化事件处理流程。
- 案例1：periodic_check 返回空列表，经查是数据库查询条件错误，修复后恢复。
- 案例2：user_message 处理后无输出，原因是消息解析失败未记录错误，增加异常捕获后解决。
- 案例3：user_message 触发后，日志仅记录 'received' 但无后续分析结果，实际是处理函数未执行解析逻辑。修复后增加对消息内容的分类和状态更新。
- 案例4：periodic_check 多次触发但结果字段为空，经查是检查函数未正确连接数据源。修复数据源连接后，日志正常输出检查结果。
